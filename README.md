## Сравнение производительности структур данных и алгоритмов
### Обзор проекта
Данный проект представляет собой комплексное исследование производительности основных структур данных и алгоритмов сортировки. 

В работе сравниваются:

#### Структуры данных:
- Хеш-таблица с цепочками для разрешения коллизий (универсальная, поддерживает любые типы данных через void*)
- Красно-чёрное дерево (сбалансированное бинарное дерево поиска)

#### Алгоритмы сортировки:
- Быстрая сортировка (Quick Sort) - O(n log n)
- Сортировка вставками (Insertion Sort) - O(n²)

Цель исследования: подтвердить теоретические оценки сложности на практике, измерить реальное время выполнения операций и выявить скрытые константы.

### Описание структур данных

#### 1. Хеш-таблица (Hash Table)

Принцип работы:
- Использует метод цепочек для разрешения коллизий
- При вставке элемента вычисляется хеш-функция, определяющая индекс в массиве
- Все элементы с одинаковым хешем попадают в одну цепочку (связанный список)
- При заполнении > 75% таблица автоматически расширяется (удваивается)

Ключевые особенности:
- Универсальность: хранит данные любого типа через указатели void*
- Динамическое расширение: автоматическое увеличение размера при переполнении
- Кастомизация: пользователь может задать свои функции хеширования, сравнения, копирования и освобождения

#### 2. Красно-чёрное дерево (Red-Black Tree)

Принцип работы:
- Самобалансирующееся бинарное дерево поиска
- Каждый узел имеет цвет (красный или чёрный)
- Поддерживаются 5 свойств, гарантирующих балансировку
- После вставки и удаления выполняются повороты и перекрашивания

Ключевые свойства:
- Каждый узел красный или чёрный
- Корень всегда чёрный
- Все листья (NULL) - чёрные
- У красного узла оба потомка чёрные
- Для каждого узла все пути к листьям содержат одинаковое количество чёрных узлов

### Описание алгоритмов сортировки
#### 1. Быстрая сортировка (Quick Sort)

Принцип работы:
- Выбор опорного элемента (pivot) - используется медиана из трёх
- Разбиение массива на элементы меньше и больше опорного
- Рекурсивная сортировка подмассивов
- Для малых подмассивов (< 20 элементов) используется сортировка вставками

Преимущества:
- Очень быстрая на практике
- Хорошая локальность данных
- Эффективная для больших массивов

#### 2. Сортировка вставками (Insertion Sort)

Принцип работы:
- Массив делится на отсортированную и неотсортированную части
- Каждый элемент из неотсортированной части вставляется на правильное место в отсортированной части
- Элементы больше текущего сдвигаются вправо

Преимущества:
- Простота реализации
- Эффективна для малых массивов (n < 100)
- Эффективна для почти отсортированных данных
- Стабильная сортировка

### Методика тестирования
Для структур данных:
- Генерация N случайных целочисленных ключей
- Измерение времени вставки N элементов
- Измерение времени поиска N элементов
- Измерение времени удаления N элементов
- Повторение тестов для разных N: 1000, 5000, 10000, 50000, 100000, 500000

Для алгоритмов сортировки:
- Генерация случайного массива размера N
- Измерение времени быстрой сортировки
- Измерение времени сортировки вставками
- Повторение тестов для разных N: 100, 500, 1000, 5000, 10000, 50000, 100000

### Анализ скрытых констант
Для каждой структуры данных вычисляется скрытая константа:

Хеш-таблица: C = T (время операции, O(1))

КЧД: C = T / log₂(n) (время / логарифм размера, O(log n))


### Результаты бенчмарков и анализ скрытых констант
### 1. Сравнение структур данных
#### 1.1 Вставка элементов
Вставка является одной из ключевых операций, определяющих производительность структур данных при построении коллекций.

#### Анализ результатов:

##### Хеш-таблица демонстрирует стабильное O(1):

- Среднее время на операцию колеблется в диапазоне 460-817 нс
- Несмотря на рост размера данных, время остаётся примерно постоянным
- Небольшие колебания связаны с состоянием кэша и операциями расширения таблицы

##### КЧД показывает логарифмический рост:

- Время на операцию растёт с 866 нс (для 1000) до 1,922 нс (для 500,000)
- Рост соответствует O(log n) - примерно в 2.2 раза при увеличении данных в 500 раз

Преимущество хеш-таблицы:

- На малых данных (1000): быстрее в 1.88x
- На средних данных (50,000): быстрее в 3.22x (максимум)
- На больших данных (500,000): быстрее в 2.35x

#### 1.2 Поиск элементов
Поиск является самой частой операцией в большинстве приложений, поэтому его производительность критична.

#### Анализ результатов:

##### Хеш-таблица - идеальное O(1):

- Время на операцию практически постоянно: 108-365 нс
- Даже при росте данных в 500 раз время увеличилось всего в 3.4 раза
- Это подтверждает теоретическую сложность O(1)

##### КЧД - чёткий O(log n):

- Время растёт с 258 до 561 нс (в 2.2 раза) при росте данных в 500 раз
- Рост соответствует логарифмической зависимости
- На 10,000 элементах показывает наибольшее отставание (4.04x)

Особенности поиска:

- Хеш-таблица особенно эффективна для поиска (до 4x быстрее)
- Пик эффективности на 10,000 элементах (ускорение 4.04x)
- На очень больших данных преимущество немного снижается из-за кэш-промахов

#### 1.3 Удаление элементов
Удаление является сложной операцией, особенно для структур с балансировкой.

#### Анализ результатов:

##### Хеш-таблица показывает O(1):

- Время на операцию: 98-667 нс
- На 500,000 элементах аномально быстро (97.66 нс) - эффект кэша
- Стабильность подтверждает O(1)

##### КЧД требует больше времени:

- Время на операцию: 181-890 нс
- Дополнительные затраты связаны с балансировкой дерева после удаления
- Ускорение хеш-таблицы: 1.33-1.90x

Особенности удаления:

- Медленнее вставки для обеих структур
- КЧД тратит время на повороты и перекрашивание
- Хеш-таблица просто удаляет узел из цепочки

### Сравнение алгоритмов сортировки
#### Ключевые наблюдения:

На малых данных (n=100):

- Сортировка вставками быстрее на 26% (0.006 мс vs 0.008 мс)
- Это связано с малой константой и отсутствием рекурсивных вызовов
- На средних данных (n=5,000):
- Быстрая сортировка уже в 16.67x быстрее
- Преимущество становится очевидным

На больших данных (n=100,000):

- Быстрая сортировка в 265,36 x быстрее!
- Разница колоссальная:  15,781 мс vs 4187,613 мс
